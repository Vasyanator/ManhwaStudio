# Module: modules/new_project

## Purpose

Python helpers for the Rust launcher "New project" flows. The active production path is Rust UI code that launches these helpers for browser automation and legacy-compatible import/download operations.

## Architecture

`adv_fetch_cli.py` (`AdvancedFetchDaemon`) and `adv_fetch_cloak_cli.py` (`CloakFetchDaemon`) provide the browser-automation logic. They are **no longer launched as separate stdio processes**: the unified AI backend hosts them in-process through `modules/ai_backend/browser/service.py` (`BrowserService`) and serves them over the framed IPC protocol (method `browser.command`). `BrowserService` instantiates the active backend's daemon class, redirects its `_emit` so `progress` events stream as IPC `progress` frames and the single terminal event becomes the IPC response, and calls the unchanged `_handle_command`. The Rust side (`src/launcher/new_project/advanced_download.rs` and `.../batch_processing/executor.rs`) sends each legacy command object as the request header `payload`; it no longer spawns Python. The `--daemon`/`main()` stdin entry point is kept only as a manual debugging fallback. The Selenium helper owns a Selenium driver; the CloakBrowser helper owns a persistent Playwright/CloakBrowser context. Both select the active page, collect image candidates from DOM and shadow DOM, filter explicit site-code resources and bare origin URLs from auto-discovery, download validated image bytes into a temporary folder, and report progress/results back to Rust. Auto-discovery uses typed DOM candidate sources so image-context URLs can keep strange extensionless shapes while obvious page/text/static resources are skipped before download. Selenium and Playwright are imported lazily on first browser use, so an AI-only backend never pays that cost. Version compatibility is checked via the `browser.version` command (was the stdio `ready` event); Rust warns when the downloader version differs from the Studio binary version.

Browser-owned state stays inside the daemon. Advanced downloader image candidates first try a parallel `requests` pass using browser-like headers and a Selenium cookie snapshot, except for `blob:` or other non-http URLs. Candidates that still fail are fetched through Selenium in the active tab, then through a temporary Selenium tab. HTTP(S) candidates are not directly navigated in that tab because some servers trigger browser downloads; browser-scoped candidates such as `blob:` may still use direct tab navigation and canvas readback. The currently successful method is preferred for the next remaining candidates in the same fetch run, but failures re-enter the fallback chain.

The CloakBrowser helper keeps the same protocol but uses Playwright page primitives. Normal fetches avoid long-lived response-body capture while the user is only browsing; they use page-context `fetch(..., credentials="include")`, DOM image canvas readback, a temporary Cloak page, and finally a `requests` fallback with context cookies for HTTP(S) URLs. Deep capture installs an observe-only init script before reloading the active chapter page. The script creates the shared capture state, assigns each canvas a stable WeakMap id, and wraps `URL.createObjectURL`, `OffscreenCanvas.convertToBlob`, and `getContext` (forcing `preserveDrawingBuffer` for WebGL) to capture DRM/descramble images; every wrapper must stay observe-only (call the native function, return its real result, swallow its own errors, install once, and spoof `toString`). It must not replace network/transport APIs such as `fetch`, XHR, or `Worker`. Response-body/CDP capture is enabled only for image-like network responses during the active capture window. Auto-review fetch commands accept an optional `cancel_file`; helpers check it between image candidates and return an `auto_result` for already saved images when cancellation is requested. Auto-review groups are computed before download using the same URL-signature shape as the Rust review window; a group with three failed candidates and no successful decoded image is rejected, and remaining links in that group are skipped.

CloakBrowser also owns a deep capture mode that is not implemented for Selenium. It copies image response bodies from the Playwright context and Chromium CDP network bodies when available, reads plain `<img>` tags straight from the page on every drain (the init-script `scanImages` helper fetches each rendered `img.currentSrc`/`src` once via page-context `fetch(credentials="include", cache="force-cache")`, covering http(s)/`blob:`/`data:` sources and authorized images, and skipping opaque cross-origin responses since the network layer already keeps those — this makes plain `<img>` pages appear in the live count instead of only at stop), polls canvas snapshots through preserved native canvas export APIs, captures `createObjectURL`/`OffscreenCanvas.convertToBlob` images drained from the init-script queue (covers DRM/descramble sites), adds visible canvas screenshots as a per-element compositor fallback, and returns decoded image bytes as the existing `auto_result` payload for Rust auto-review. Stop drains a few extra times (`_settle_deep_image_reads`) so in-flight `<img>` reads from a fast start→stop still land. The `deep_intercept_status` poll reports a live breakdown rather than one number: the `intercept_count` event carries `found_pages` (raw payload total), `found_canvases` (canvas-element captures), and `found_images` (everything else — plain `<img>`, network bytes, blob/descramble exports), classified by `_deep_capture_is_canvas_source`, so the UI can show "captured N canvases, found M regular images". These are raw capture counts, not the deduped/clustered page count produced at stop.

CloakBrowser canvas fetch/intercept emits detailed JSON `log` events while inspecting and saving canvases. Those logs include DOM backing size, CSS box size, viewport visibility, export errors, data URL shape, decoded PNG size, byte size, and simple black/transparent pixel diagnostics; Rust forwards them to the runtime log for troubleshooting pages that export blank or low-quality canvases.

## Files and submodules

- `adv_fetch_cli.py`: Selenium daemon for open-url, candidate fetch, link collection, canvas fetch, and canvas intercept commands.
- `adv_fetch_cloak_cli.py`: CloakBrowser/Playwright daemon with the same Rust-facing command protocol, deep capture mode, and persistent `modules/browser_profiles/cloak_profile` profile.
- `test_adv_fetch_cloak_cli.py`: unit tests for Cloak deep-capture ordering, blank detection, and duplicate-collapse contracts that do not require a browser.
- `common.py`: shared helpers for URL/pattern handling used by new-project download flows.
- `downloaders.py`: legacy PyQt-facing wrappers for downloader actions.
- `batch_nodes_window/`: Python legacy batch-node UI/runtime retained for compatibility.
- `window.py`: legacy Python new-project window.

## Contracts and invariants

- `_handle_command` is the single command contract reused by both transports: it takes one command dict and reports exactly one terminal event (plus interim `progress` events) through `self._emit`. `BrowserService` depends on this shape, so it must be preserved.
- Browser commands run inside the always-on AI backend, serialized by `BrowserService`'s lock; selenium/playwright imports must stay lazy so AI-only startup never loads them.
- Version compatibility is reported via the `version` command (`downloader_version`); the legacy `ready` stdout event only matters for the manual `--daemon` fallback.
- Selenium driver calls must remain in the daemon-owned browser context. Browser-tab fallbacks are serialized because WebDriver is not thread-safe; the requests pass may use `max_parallel`.
- CloakBrowser page calls must remain serialized through the helper's page lock; background collection loops may poll status data, but they must not run concurrent Playwright page operations against the same context.
- CloakBrowser uses Playwright's sync API; browser callbacks and Python background threads must not call Playwright methods directly. They may only enqueue lightweight state for the daemon command thread to drain during status/stop/fetch commands.
- Progress events should remain monotonic for a fetch run.
- Temporary output folders contain only successfully decoded images, named in page order.
- Auto-review cancellation is cooperative and must preserve already decoded images rather than killing the daemon.
- Auto-review downloads must reject redirects and text/page-like URL suffixes while still accepting extensionless or otherwise unusual URLs when the returned bytes decode as supported image data; misleading HTTP content types must not override byte-level image validation.
- Auto-review group rejection is pre-download and group-scoped: failed candidates are counted before the Rust review step so unproductive URL families stop being attempted.
- Cloak deep capture must reload the active page after installing init hooks; output classification must remain byte-based and feed the same Rust auto-review contract instead of bypassing the review window. The active page is the tab the user has in front, picked by `_select_active_page`: a focused tab wins, then a visible tab, and only when the window is hidden/occluded (neither readable) the most recently opened tab; each group resolves to its newest member via `_page_open_order` (tracked from the context `page` event), so a chapter opened in a new tab is reloaded rather than the first-opened list tab. Only that one tab is reloaded.
- Cloak deep capture auto-review items use one synthetic `deep-capture://capture/page/*.png` URL family so captures from network, blob, data, and canvas sources stay in one review group; source URLs are diagnostic metadata, not grouping keys. Finalization is a content pipeline: blank (near-uniform black/white) frames are dropped globally so off-screen unrendered canvases never become pages; remaining frames are collapsed to the best frame per element (keyed by stable WeakMap id, then `dom_order`); records are clustered by perceptual hash so one page captured through several layers is one page; the highest-fidelity representative wins per cluster; pages are ordered by their position in page document order (the live stop-time DOM order with pages that already scrolled out of the DOM prepended in first-seen order, the reliable reading order for plain-`<img>` sites), then by per-snapshot DOM-walk order, on-page geometry, URL-embedded page numbers, and finally capture order; size-outlier pages are flagged `probable_junk` and the review window opens with them unchecked.
- The cumulative deep-capture DOM order (`_accumulate_deep_dom_order`) is refreshed on every drain by plain first-seen append (`_append_first_seen_keys`): each poll's window of image-URL/canvas-WeakMap-id keys is appended in encounter order. Anchor-based insertion must not be used — consecutive polls often catch non-overlapping windows during a fast scroll, and inserting a non-overlapping group relative to absent anchors scrambles the sequence into small correctly-ordered fragments. At stop, `_combine_dom_order` prepends only the keys that have left the DOM (in first-seen order) to the live stop-time DOM order, so pages still on the page keep the reliable stop order (identical to a single stop-time read) while early pages recycled out of a virtual-scroll reader keep their reading position.
- Browser sessions are cached and reused across commands, but `_ensure_browser` must verify the cached session is still live and relaunch when it is not: any failure probing the Selenium driver (closed window, dead chromedriver, invalid session) drops it, and a CloakBrowser context with no live page (the user closed the browser) is torn down and recreated. A command like `open_url` must always end up with a live browser instead of erroring on a stale one.
- Errors returned to Rust must distinguish user-facing messages from technical log messages.

## Editing map

- To change advanced browser image extraction, edit `adv_fetch_cli.py`.
- To change wildcard/prefix matching shared by download flows, edit `common.py`.
- To change legacy Python UI glue, edit `downloaders.py` or `window.py`.
